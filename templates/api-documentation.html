Anomaly Detection Model API: Documentation

Introduction

The Anomaly Detection Model API is designed to identify abnormal behaviour in subscription or licensing-based software systems. Industries that operate on subscription or licensing models often face challenges such as account sharing, multiple unauthorized usage, and abnormal system behaviour. Traditional rule-based detection systems are often inaccurate and struggle to identify complex patterns of misuse.

This API analyzes incoming data and identifies unusual behaviour based on patterns learned from historical data. The system evaluates requests and produces a set of analytical outputs including anomaly score, risk indicators, and model confidence. By integrating this API, developers can automatically detect suspicious activity and take preventive actions to protect revenue, improve customer retention, and ensure fair usage of subscription or licensed services. Below is the public URL to access the API:

https://anomalydetectionmodel-api.onrender.com/

API Endpoints

Here are available Public APIs:

Method Endpoint Description
GET / Home
POST /predict Send input data to the ML model embedded api and receive anomaly prediction results
GET /docs Open the Swagger UI interactive API documentation (Try API Here !!!)
GET /documentation View the complete API documentation webpage
GET /model-metadata Retrieve model information

Request Format

Note: All API requests except the POST /predict endpoint return a webpage response.

To obtain anomaly predictions from the model, send a POST request to the /predict endpoint with the required input features in JSON format.

Request Body (JSON)

{
  "devices_count": int,
  "avg_session_duration": float,
  "api_rate": float,
  "geo_jump_km": float,
  "activations_24h": int,
  "failed_login_ratio": float,
  "api_std_7d": float,
  "session_trend": int
}

Response Format

A successful request to the /predict endpoint returns a JSON response with the following fields:

Response Body (JSON)

{
    "Score": float (0 to 1 Value),
    "Label": int (-1 : Anomaly or 1 : Normal),
    "Risk_Level": str (Low, Medium, High),
    "Model_Confidence": float (0 to 1 Value),
    "Model_Agreement": int (0 : Disagree or 1 : Agree),
    "Action": str (e.g., M10, M20, M30)
}

Field Explanation

Here is an explanation of each field in the request and response:

Request Fields:

Field Description
Device Count Total number of devices associated with the user or account during the observation period.
Average Session Duration Average time (in minutes or seconds) a user session remains active.
API Rate Number of API requests generated within a specific time window.
Geo Jump (km) Distance between two consecutive login locations, measured in kilometers.
Activations 24h Total number of account or license activations recorded within the last 24 hours.
Failed Login Ratio Ratio of failed login attempts compared to total login attempts.
API Std 7d Standard deviation of API request activity over the last 7 days.
Session Trend Trend indicator showing whether session activity is increasing, decreasing, or stable.

Response Fields:

Field Description
Score (Anomaly Score) A numerical value between 0 and 1 indicating how abnormal the input behaviour is. Higher values represent more anomalous activity.
Label (Anomaly Label) Model classification result indicating whether the input is Normal (1) or Anomalous (-1).
Risk Level Categorical risk assessment derived from the anomaly score, typically classified as Low, Medium, or High risk.
Model Confidence A value between 0 and 1 representing how confident the models is in its prediction.
Model Agreement Indicates whether the ensemble models agree on the prediction (1 = agreement, 0 = disagreement).
Action Recommended action based on the prediction, such as Allow, Monitor, or Investigate.

Action Field:

Action Code Risk Level Description
A00 Low No suspicious behaviour detected. The user activity is considered normal and access can be allowed.
M10 Low – Medium Minor anomalies detected. The user should be monitored for unusual behaviour.
V20 Medium Suspicious activity detected. The system should verify the user through additional authentication.
O30 Medium – High Strong anomaly signals detected. Request additional verification or identity confirmation before allowing access.
B40 High High risk anomaly detected. Access should be blocked to prevent potential misuse.

Sample Request

Sample User Data:

Method : POST  Endpoint: /predict
Content-Type: application/json
{
  "devices_count": 3,
  "avg_session_duration": 42.5,
  "api_rate": 18.2,
  "geo_jump_km": 1200.4,
  "activations_24h": 4,
  "failed_login_ratio": 0.82,
  "api_std_7d": 5.8,
  "session_trend": 1
}

API Response:

{
  "Score": 0.78,
  "Label": -1,
  "Risk_Level": "High",
  "Model_Confidence": 0.91,
  "Model_Agreement": 1,
  "Action": "B40"
}
Explanation: According to the request, the API responds with the JSON output shown in the block above. The Score indicates how unusual the behavior is, while the Label classifies the user as anomalous.Based on this prediction, the system assigns a High Risk Level and recommends the action B40.

Error Response:

If the request is invalid, missing required fields, or contains incorrect data types, the API will return an error response.

{
  "error": "Invalid request",
  "message": "Missing required field: devices_count"
}

HTTP Status Codes

Status Code Description
200 Request successful. Prediction returned.
400 Invalid request or missing required parameters.
422 Invalid data format or incorrect data type.
500 Internal server error.
